## 1. Что это за продукт

B2B-инструмент для технологов обогатительных фабрик. Вход: Excel-отчёт
института по анализу хвостов (отвальных отходов флотации), схема/регламент
фабрики, база знаний (учебники PDF). Выход: ранжированный список проверяемых
гипотез по снижению потерь никеля и меди в хвостах — каждая с механизмом,
цитатами из литературы, эффектом в тоннах/деньгах, рисками и планом проверки.
Плюс экспорт в DOCX/CSV/JSON и фидбэк-луп (принять/отклонить гипотезу).

Ключевые требования жюри: интерпретируемость (почему предложена гипотеза),
работа с шумными данными, цитируемость источников, оценка эффекта на KPI,
масштабируемость на другие домены, локальное развёртывание.

## 2. Доменный контекст (обязателен для промптов и правил)

Флотация: измельчённую руду смешивают с водой (пульпа), реагенты делают
сульфидные минералы гидрофобными, воздушные пузырьки поднимают их в пену
(концентрат), порода тонет (хвосты). Металл в хвостах = потери.

Рабочее окно флотации ~10–100 мкм. Потери концентрируются по краям:
- Крупные классы (+71, +125 мкм) + «закрытый» минерал (сросток с породой,
  пузырёк не видит его поверхность) = НЕДОИЗМЕЛЬЧЕНИЕ. Лечится в переделах
  измельчение/классификация: футеровка и шары мельниц, насадки гидроциклонов,
  тонкое грохочение, контроль гранулометрии.
- Тонкий класс (−10 мкм) + «раскрытый» минерал = ПЕРЕИЗМЕЛЬЧЕНИЕ/ШЛАМЫ.
  Лечится: флотация (время, фронт, плотность пульпы, реагенты, контактные
  чаны) + классификация (не перемалывать готовое).
- Раскрытый минерал в средних классах = недоработка самой флотации.

Минералогия (критично для логики):
- Элемент 28 = никель (Ni), Элемент 29 = медь (Cu) — в данных анонимизированы.
- ИЗВЛЕКАЕМЫЕ формы для Ni: раскрытый Pnt/Cp, закрытый Pnt/Cp, миллерит.
  Для Cu: только раскрытый и закрытый Pnt/Cp. (Pnt = пентландит, Cp = халькопирит.)
- НЕИЗВЛЕКАЕМЫЕ: примесь в пирротине (Ni в кристаллической решётке),
  силикатная форма/валлериит, пирит. По ним гипотезы НЕ ГЕНЕРИРУЮТСЯ —
  система явно объясняет почему (раздел «Почему не предложено»).

Фабрики в данных: КГМК (Кольская), НОФ (Норильская, ветки вкрапленных «вкр»
и медистых «мед» руд), ТОФ (Талнахская — единственная с двумя типами хвостов:
породные и пирротиновые).

Переделы фабрики (пространство гипотез): дробление → измельчение →
классификация → флотация (основная/контрольная/перечистная) → реагентика →
вспомогательные (магнитная сепарация, гравитация, тонкое грохочение).
Гипотезы про флотацию — меньшинство; большинство экспертных гипотез — про
измельчение и классификацию (см. эталоны).

## 6. Каноническая модель данных (pydantic, backend/app/models.py)

```python
class MineralForm(str, Enum):
    OPEN_PNT_CP = "Раскрытый Pnt/Cp"; CLOSED_PNT_CP = "Закрытый Pnt/Cp"
    PYRRHOTITE_IMPURITY = "Примесь в пирротине"; SILICATE = "Силикатная форма"
    PYRITE = "Пирит"; MILLERITE = "Миллерит"

RECOVERABLE = {  # достижимость по элементам
  "Ni": {OPEN_PNT_CP, CLOSED_PNT_CP, MILLERITE},
  "Cu": {OPEN_PNT_CP, CLOSED_PNT_CP},
}

class LossCell(BaseModel):        # универсальная ячейка карты потерь
    axes: dict[str, str]          # {"size_class": "+125", "mineral_form": "Закрытый Pnt/Cp"}
    element: Literal["Ni","Cu"]
    tonnes: float
    recoverable: bool
    process_area: str | None      # "измельчение" | "классификация" | "флотация" | None
    provenance: Literal["measured","recovered_math","recovered_llm","manual"] = "measured"
    # measured — из файла; recovered_* — восстановлено (раздел 7.1); manual — правка юзера

class TailingsReport(BaseModel):
    plant: str; period: str | None
    feed_tonnes: float; tails_tonnes: float
    grade: dict[str, float]       # {"Ni": 0.1004, "Cu": 0.0344} в %
    losses_tonnes: dict[str, float]
    size_classes: list[SizeClassRow]
    cells: list[LossCell]
    recoverable_total: dict[str, float]
    issues: list[DataIssue]       # #REF!, несходящиеся суммы, аномалии

class Hypothesis(BaseModel):
    id: str; title: str
    process_area: str             # передел
    target_cells: list[dict]      # какие ячейки карты лечит + тонны
    mechanism: str                # 2-4 предложения физики
    rationale: list[Citation]     # >=1; Citation{quote<=40 слов, source, page, chunk_id, verified: bool}
    equipment: list[EquipmentRef] # {name, positions[], present_on_plant: bool}
    effect: Effect                # {tonnes_max, tonnes_expected, money_usd, assumptions: str}
    risks: list[str]
    feasibility: dict             # {capex: low/med/high, downtime_hours, complexity}
    novelty: dict                 # {score 0-1, prior_matches: [title эталонной, если совпала]}
    verification_plan: list[Step] # Step{n, action, resources, duration, success_criterion, fail_criterion}
    score: float                  # итог ранжирования
    status: str                   # proposed/accepted/rejected/testing/confirmed/refuted
```

Эффект: tonnes_max = сумма извлекаемых тонн целевых ячеек; tonnes_expected =
tonnes_max × capture_rate (детерминированный коэффициент по типу гипотезы из
domain_packs/flotation.yaml, напр. доизмельчение 0.2–0.35, реагентика 0.1–0.2);
money_usd = tonnes_expected × 16500 (Ni) / 9000 (Cu) $/т, константы в конфиге.

## 7. Правила диагностики (diagnostics.py — детерминированный код, БЕЗ LLM)

R1 недораскрытие: суммарные потери извлекаемого металла в классах +71 и +125
   > 25% всех потерь И доля закрытого Pnt/Cp в этих классах > 40% →
   диагноз(зона="измельчение/классификация", cells=[...], tonnes=...).
R2 шламы: потери в классе −10 > 20% всех И доля раскрытого Pnt/Cp в нём > 40%
   → диагноз(зона="флотация+классификация", причина="переизмельчение").
R3 недоработка флотации: раскрытый Pnt/Cp в классах −71+45, −45+20, −20+10
   суммарно > 15% потерь → диагноз(зона="флотация: время/реагенты/плотность").
R4 потолок: посчитать неизвлекаемое по каждой форме → раздел «Почему не
   предложено» с тоннами и причиной.
R5 аномалии данных: (а) суммы «Итого (проверка)» отличаются от 100% более чем
   на 0.5% или тонны не бьются с итогом >1%; (б) #REF!/пустые в минералогии;
   (в) >50% потерь класса в неожиданной форме (пирит, силикаты для тонкого
   класса) → issue(severity, message="проверьте пробу/анализ", cell).
Каждый диагноз хранит: правило-ID, входные числа, текст на русском. Это и есть
интерпретируемость — правило показывается в UI по клику.

## 7.1 Восстановление битых данных (parser/recover.py)

Битые значения (#REF!, пустые, NaN — как в минералогии Примера 1) не
блокируют пайплайн: система пытается их восстановить, помечает происхождение
и отдаёт человеку на перепроверку. Два яруса, строго в этом порядке:

Ярус 1 — детерминированное восстановление по балансовым уравнениям (без LLM).
Отчёт переопределён: почти каждое число выводится из соседних. Инварианты:
- доли форм внутри класса суммируются в 100%; тонны форм — в тонны класса;
- тонны класса = потери элемента × доля класса; доли классов → 100%;
- извлекаемое класса = раскрытый + закрытый Pnt/Cp (+ миллерит для Ni);
- итог по классам = потери элемента в хвостах = хвосты СМТ × содержание.
Решатель: собрать систему инвариантов, найти битые ячейки, восстанавливать
итеративно те, где не хватает ровно одного неизвестного (propagation), до
неподвижной точки. Восстановленное: provenance="recovered_math",
в issue пишется формула («100% − сумма остальных форм = 57.48%»).

Ярус 2 — LLM-оценка (FAST-модель), только для ячеек, не закрытых ярусом 1.
В промпт: аналогичные блоки из других классов ЭТОГО отчёта + типовые
распределения из трёх остальных примеров (захардкодить как справочник в
domain_packs/flotation.yaml: типичный диапазон доли каждой формы по классам).
Модель возвращает значение + диапазон уверенности + объяснение.
provenance="recovered_llm", confidence сохраняется.

Правила поверх обоих ярусов:
- восстановленные значения НИКОГДА не участвуют в контрольных суммах как
  «истинные» — валидатор проверяет только measured;
- диагнозы, опирающиеся на recovered_llm-ячейки, получают флаг
  uncertain=true и показываются с оговоркой; на recovered_math — норм;
- эффект гипотезы, посчитанный на восстановленных данных, в карточке помечается
  «оценка на восстановленных данных».

UI (экран 1): восстановленные ячейки — янтарная заливка + пунктирная рамка,
тултип «Восстановлено: <формула|LLM-оценка, confidence>. Проверьте вручную»,
клик → inline-редактирование (после правки provenance="manual", подсветка
снимается). Счётчик в шапке: «Восстановлено 12 значений — проверьте».
Кнопка «Принять все восстановленные» для быстрого пути.

Тест (обязательный, P1): Пример 1 — ярус 1 восстанавливает минералогию из
инвариантов там, где хватает данных; ни одно #REF! не доходит до диагностики
как NaN; все восстановленные имеют provenance != "measured"; на Примере 2
(целом) решатель не «восстанавливает» ничего (0 срабатываний).

## 8. Генерация гипотез (hypotheses/generate.py)

Пайплайн: диагнозы R1–R3 + топ-K чанков из KB (гибридный поиск по запросам,
составленным из диагноза: «доизмельчение сростков пентландит», «флотация
шламов реагентный режим» и т.п.) + онтология оборудования фабрики +
ограничения пользователя + история прежних гипотез → один вызов STRONG-модели
с response_format=json, схема = list[Hypothesis без score/status].

Требования к промпту (хранить в hypotheses/prompts.py, на русском):
- роль: главный обогатитель НИИ; гипотезы конкретные и проверяемые в лабе;
- на каждый диагноз 2–4 гипотезы, суммарно 8–14;
- ЦИТАТЫ строго из переданных чанков, с chunk_id; выдумывать нельзя;
- гипотезы по неизвлекаемым формам запрещены;
- оборудование упоминать только из переданной онтологии, иначе ставить
  equipment.present_on_plant=false и capex=high;
- план проверки по шаблону: лаб. тест → ОПИ (опытная линия vs контрольная) →
  тираж; у каждого шага числовой критерий успеха/провала.
После генерации: verify.py прогоняет каждую цитату fuzzy-матчем (rapidfuzz,
partial_ratio > 75) по тексту chunk_id; не сошлось → verified=false, в UI
бейдж «требует проверки». Гипотезы без единой verified-цитаты понижаются в score.

Онтология оборудования: domain_packs/flotation.yaml, секция equipment —
захардкодь из списка оборудования кейса: мельницы МШЦ 4,5×6,0 (поз. 4-2, 5-3,
5-5), МШРГУ 4,5×6,0 (4-3, 5-1, 5-2, 5-4), МШР 3,2×3,8 (4-5, 4-11),
классификаторы спиральные 1КСП24/2КСН24, гидроциклоны ГЦ-660 (4-2, 5-3, 5-5),
флотомашины ФПМ-16-4К (18+11 шт), ФМ-16УМ-4К (4), дробилки КРД 700/75,
КМД-2200Т, грохоты Sibra 2DR, сгустители П-30/П-50, насосы. Плюс маппинг
«тип гипотезы → требуемое оборудование».

Ранжирование (rank.py): score = w1*норм(effect.money) + w2*(1-capex_норм) +
w3*(1-risk_норм) + w4*novelty. Веса из запроса (слайдеры UI), дефолт
0.4/0.25/0.2/0.15. Novelty: эмбеддинг-близость к эталонным гипотезам и
принятым ранее: cos>0.8 → prior_match, novelty=0.2.

Фидбэк: POST /hypotheses/{id}/feedback {action: accept/reject, reason} →
пишется в SQLite; reject c reason добавляет направление в стоп-лист проекта,
регенерация исключает его (передаётся в промпт).

## 8.1 Чат-интерпретатор (backend/app/chat.py)

Контекстный ассистент проекта: отвечает на вопросы про ЭТОТ отчёт, диагнозы
и гипотезы. Не general-purpose чат, а слой интерпретации поверх пайплайна.
Типовые вопросы: «почему гипотеза про насадки выше футеровки?», «откуда
846 т?», «что такое закрытый Pnt/Cp?», «что если убрать ограничение по CAPEX?».

Устройство: POST /projects/{id}/chat {message, history[]} → FAST-модель
(STRONG только если history длинная или вопрос про сравнение гипотез).
Контекст собирается детерминированно перед вызовом:
- TailingsReport (компактный JSON: баланс, классы, топ-ячейки, issues);
- все диагнозы с правилами-ID и входными числами;
- заголовки+эффекты+score всех гипотез, полные карточки — только упомянутых
  в вопросе (матчинг по id/заголовку);
- топ-5 чанков KB по вопросу (тот же kb/search);
- стоп-лист и веса проекта.
Правила промпта: отвечать ТОЛЬКО из переданного контекста; каждое число —
со ссылкой на источник в формате [правило R1] / [ячейка +125/закрытый Pnt/Cp]
/ [Глембоцкий, с. N]; нет ответа в контексте — сказать прямо и предложить,
куда посмотреть; терминологию объяснять на уровне инженера, по-русски.
Ответ: {text, references: [{type: rule|cell|hypothesis|chunk, id}]} —
фронт рендерит ссылки кликабельными (скролл к правилу/карточке, попап чанка).

UI: выдвижная панель справа, доступна на всех экранах (иконка в шапке),
контекст проекта подтягивается автоматически. 3 стартовые подсказки-чипса:
«Объясни главный диагноз», «Почему гипотеза №1 первая?», «Что неизвлекаемо и почему?».

Тест (мокнутый LLM): контекст-сборщик для вопроса «откуда 846 т» включает
ячейку +125/закрытый Pnt/Cp и правило R1; ответ содержит references.

## 8.2 Дорожная карта проверки (hypotheses/roadmap.py)

Строится по кнопке для гипотез со статусом accepted. Два уровня.

Уровень гипотезы: verification_plan карточки разворачивается в стадии с
датами. Шаблон стадий (длительности по типу гипотезы из
domain_packs/flotation.yaml, поле verification_profile):
лабораторные испытания (2–4 нед) → ОПИ на опытной линии vs контрольная
(4–8 нед) → тираж + корректировка регламента (2–4 нед). Между стадиями —
ворота: числовой критерий успеха из плана; исход провала = гипотеза
закрывается, фиксируется в фидбэк. Дешёвые обратимые вмешательства (насадки,
реагенты) могут пропускать лабу — флаг skip_lab в профиле.

Уровень портфеля (Гант): все принятые гипотезы на одной шкале времени
(месяцы/кварталы, горизонт до 2 лет). Планировщик — детерминированный код:
- КОНФЛИКТ РЕСУРСОВ: у каждого типа гипотезы в yaml прописан occupied_resource
  (мельница 5-3, секция флотации, лаборатория). Две гипотезы с одним ресурсом
  не пересекаются по времени — вторая сдвигается (иначе эффекты не разделить);
  независимые переделы идут параллельно.
- Порядок в очереди на общий ресурс = score гипотезы (сначала выгодные).
- Ёмкость лаборатории: max 2 одновременных лаб. программы (конфиг).
Выход: RoadmapItem{hypothesis_id, stage, start, end, resource, gate_criterion,
depends_on}. Пересчёт при изменении состава принятых гипотез.

API: POST /projects/{id}/roadmap/build → Roadmap;
GET /projects/{id}/roadmap; PATCH /roadmap/items/{id} {start} — ручной сдвиг
(планировщик валидирует конфликты, при нарушении возвращает 409 с причиной).

UI (экран 4, вкладка «Дорожная карта»): Гант — строки=гипотезы, сегменты=
стадии (лаба/ОПИ/тираж разными оттенками), ромбы=ворота с критерием в
тултипе, вертикальная линия «сегодня», шкала по кварталам, конфликт-сдвиги
показаны серым пунктиром «ждёт мельницу 5-3». Клик по строке → раскрытие
плана гипотезы. Рендер — свой на CSS grid или лёгкая либа (frappe-gantt),
не тащить тяжёлые chart-киты. Экспорт: стадии попадают в tasks.csv
(колонки start/end/resource/criterion) и отдельным листом в DOCX-отчёт.

Тест: 3 фиктивные принятые гипотезы, две конфликтуют по мельнице 5-3 →
планировщик разводит их последовательно по score, третья (флотация)
параллельна; PATCH со сдвигом в конфликт → 409.

## 9. API и экраны

API (FastAPI, все ответы pydantic):
- POST /projects {plant, goal, constraints} → project
- POST /projects/{id}/report (upload xlsx) → TailingsReport c issues
- PATCH /projects/{id}/report/cells — ручные исправления
- GET  /projects/{id}/diagnostics → диагнозы + карта потерь
- POST /projects/{id}/hypotheses/generate {weights, excluded_areas} → list[Hypothesis]
- POST /hypotheses/{id}/feedback
- GET  /projects/{id}/export/docx | /export/tasks.csv | /export/json
- POST /kb/upload (pdf) → {status: indexed | scan_no_text, pages, chunks}
- GET  /kb/documents; GET /kb/chunk/{id} (для показа цитаты)
- POST /kb/ask {question} → ответ с цитатами (для демо второго домена: золото)
- POST /projects/{id}/chat {message, history} → {text, references[]} (раздел 8.1)
- POST /projects/{id}/roadmap/build; GET .../roadmap; PATCH /roadmap/items/{id} (раздел 8.2)

Экраны (React, роутер, шапка проекта со степпером Данные → Диагностика →
Гипотезы → Отчёт):
1. «Разбор отчёта»: сводка карточками (тоннаж, потери Ni/Cu, % извлекаемого),
   таблицы крупности и минералогии, строки с issues подсвечены янтарным с
   бейджем и inline-редактированием, восстановленные ячейки по 7.1 (пунктирная
   рамка + тултип с формулой/confidence + счётчик «Восстановлено N — проверьте»),
   кнопка «Подтвердить».
2. «Карта потерь»: тепловая матрица классы×формы (тонны, интенсивность=величина),
   колонки неизвлекаемых приглушены с замочком, переключатель Ni/Cu, справа
   панель диагнозов (карточка = текст + правило по клику + CTA
   «Сгенерировать гипотезы (извлекаемо N т)»). Блок «Почему не предложено».
3. «Гипотезы»: слева слайдеры весов + фильтры по переделам + тумблер «без
   капзатрат» + поле исключений; справа карточки: ранг, заголовок, теги
   (передел, CAPEX, риск), «до N т Ni/год ≈ $M», скоринг-бар; раскрытая:
   механизм, цитаты-чипсы (клик → фрагмент источника, бейдж verified),
   мини-карта целевых ячеек, чипсы оборудования «✓ есть на фабрике»,
   риски, план проверки таймлайном с критериями; футер: Принять/Отклонить(причина)/
   Комментарий. Совпавшие с эталоном — зелёный бейдж «Совпадает с гипотезой экспертов».
4. «Отчёт и экспорт»: две вкладки. «Отчёт» — превью DOCX (титул, сводка,
   топ-5 таблицей), кнопки DOCX/CSV/JSON, канбан статусов принятых гипотез.
   «Дорожная карта» — Гант из 8.2.
5. «База знаний»: список документов со статусами (indexed / scan_no_text →
   бейдж «требуется OCR»), дропзона, поле вопроса (ask) с цитируемым ответом.
6. Чат-панель (раздел 8.1): выдвижная справа, на всех экранах, кликабельные
   references, чипсы-подсказки.

Если фронт не успевает целиком — приоритет экранов: 3 → 2 → 6 → 1 → 5 → 4;
недостающее закрыть работающими API + разделом в README с curl-примерами.

## 10. Эвал (eval/run_eval.py) — обязательная фаза

Для каждого из 4 примеров: парсинг xlsx → диагностика → генерация → сравнение
с эталонным docx. Матчинг: FAST-модель как judge — «покрывает ли
сгенерированная гипотеза X эталонную Y (same intervention)?» да/нет.
Метрики в eval/report.md:
- coverage: доля эталонных гипотез, воспроизведённых системой (цель ≥ 60%);
- novel: число валидных новых гипотез;
- citation_validity: доля verified-цитат (цель ≥ 90%);
- parse_ok: все 4 файла распарсены без падений (Пример 1 — с issues, Пример 4 —
  оба типа хвостов).
Прогнать минимум дважды, результаты в отчёт. Это материал для слайда.

## 12. Правила автономной работы (loop)

1. Порядок фаз: P0 скаффолд+CI(pytest) → P1 парсер xlsx+docx + восстановление
   данных (7.1) → P2 диагностика → P3 KB/RAG → P4 генерация+verify →
   P5 ранжирование+фидбэк+store → P6 чат (8.1) + дорожная карта (8.2) +
   экспорт → P7 фронтенд → P8 eval → P9 demo_seed+README. Не начинай
   следующую фазу, пока тесты текущей не зелёные.
2. Каждая фаза = коммит с осмысленным сообщением. TODO.md обновляй после
   каждой фазы: done / blocked / deferred.
3. Тесты пиши ДО или ВМЕСТЕ с кодом фазы. Минимум: P1 — парсинг всех 4 xlsx с
   проверкой контрольных чисел из раздела 3 (допуск 0.01), восстановление
   Примера 1 по 7.1 (и 0 ложных срабатываний на Примере 2); P2 — на Примере 2
   срабатывают R1 (закрытый Pnt/Cp в +125 = 845.73 т) и R2 (раскрытый в −10),
   R5 ловит аномалию Cu/пирит; P4 — схема JSON валидна, цитаты верифицируются
   на подложенной фикстуре; P6 — планировщик Ганта разводит конфликт ресурса
   (тест из 8.2); P8 — eval отрабатывает.
4. LLM-вызовы в тестах мокай (фикстуры ответов в tests/fixtures/). Живые
   вызовы — только в eval и demo_seed, и только если есть LLM_API_KEY.
5. Застрял на чём-то > 30 минут — сделай заглушку с TODO-комментарием, запиши
   в TODO.md и иди дальше. Приоритет: работающий end-to-end путь Примера 2
   важнее любой отдельной фичи.
6. Эмбеддинг-модель качается долго — начни загрузку в фоне в начале P3; если
   недоступна, переключись на fallback из env и зафиксируй в TODO.md.
7. Ничего не удаляй и не переименовывай в data/case/. Имена файлов содержат
   кириллицу и NFD/NFC-нюансы — в коде обходи директории через os.scandir,
   не хардкодь пути строками, нормализуй unicodedata.normalize("NFC", name).
8. Не строй: авторизацию, докер-оркестрацию, i18n, тёмную тему. Чат — только
   как панель-интерпретатор из 8.1, НЕ как основной интерфейс продукта: главный
   UX — структурированные экраны, LLM спрятана внутри процесса.
8а. Индекс KB хранит имя эмбеддинг-модели; при смене EMBED_MODEL — авто-реиндекс,
   а не тихая выдача мусора из-за несовпадения размерностей.
9. Финальный чеклист перед остановкой: pytest зелёный; demo_seed отрабатывает
   с ключом и без (мок); README-команды воспроизводимы с нуля; eval/report.md
   существует; git log читаем.
