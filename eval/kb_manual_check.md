## Предложение фильтра служебных чанков (НЕ применено — только патч-идея)

```python
# kb/chunking.py — кандидат: отсев/усечение служебных чанков перед выдачей
SERVICE_MARKERS = re.compile(
    r"^\s*(СПИСОК ЛИТЕРАТУРЫ|БИБЛИОГРАФИЧЕСКИЙ СПИСОК|СОДЕРЖАНИЕ|ОГЛАВЛЕНИЕ|REFERENCES)",
    re.IGNORECASE | re.MULTILINE)

def is_service_chunk(text: str) -> bool:
    if SERVICE_MARKERS.search(text[:300]):
        return True                                   # титулы разделов-справочников
    digits = sum(c.isdigit() for c in text) / max(len(text), 1)
    if digits > 0.22:
        return True                                   # таблицы/шапки таблиц
    lines = [l for l in text.splitlines() if l.strip()]
    short = sum(1 for l in lines if len(l.strip()) < 25)
    if lines and short / len(lines) > 0.6:
        return True                                   # оглавления, столбцы таблиц
    captions = len(re.findall(r"\b(Рис\.|Табл(?:ица)?\.?|Fig\.)", text))
    return captions * 1000 / max(len(text), 1) > 3    # сплошные подписи к рисункам

# + усечение хвоста: text = SERVICE_MARKERS.split(text)[0] — чтобы «СПИСОК ЛИТЕРАТУРЫ»
# не съедал последний содержательный чанк статьи (кейс bolobov:0005)
```

Оценка эффекта: убрал бы 5 из 8 замеченных мусорных попаданий (шапка патента —
отдельный случай: у патентных txt первую сотню строк с перечнем авторов лучше
резать на этапе fetch).

## Вердикт

Индекс пригоден как источник цитат для генератора **с оговоркой**: чистый dense-256
уверенно закрывает «широкие» темы (футеровка, депрессия пирротина, грохочение —
цитаты вырезаются дословно и адресно), но проваливает лексически-точные запросы:
«насадки гидроциклонов», «потери со шламами», «время контрольной флотации» дают
общесмысловых соседей вместо точного термина. Именно этот класс промахов закрывает
BM25 в гибридном поиске основного приложения — термины «насадка», «шламы», «контрольная
флотация» в корпусе лексически присутствуют (Пелевин, Глембоцкий §7 «Кинетика», Усманова),
BM25 их найдёт по точному совпадению, а RRF смешает с dense-выдачей. Останутся слабыми
даже в гибриде: (1) «замена насадок ГЦ» — материала о влиянии диаметра насадки на слив
в корпусе, похоже, нет вовсе (кандидат на пополнение корпуса методичкой по гидроциклонам);
(2) OCR-битые формулы кинетики у Глембоцкого — цитата есть по смыслу, но текст нечитаем.
Служебный мусор (колонтитулы, подписи рисунков, шапки таблиц, списки литературы) занимает
~15% топов и лечится фильтром выше без переиндексации всего — только пересборкой чанков.

## Аддендум 2: применённые улучшения (фильтр мусора + гибрид BM25+dense)

По итогам валидации применены оба предложения (ранее лежали как патч-идеи):

1. **Фильтр служебных чанков** в `kb/chunking.py` (`is_service_chunk` +
   `trim_service_tail`): корпус 1146 → 1052 чанка (−94, в основном патентная мета
   Google Patents с хэшами веществ, таблицы, титулы разделов, библиография срезается
   с хвостов). Пороги калиброваны на корпусе: доля цифр > 0.25, короткие строки
   > 80% при ≥8 строках (первый вариант 0.6 срезал инженерный текст с формулами —
   пойман и исправлен), плотность подписей «Рис./Табл.». Все цитаты-кандидаты из
   таблицы выше пережили фильтр (проверено пробами по подстрокам).
2. **Гибридный поиск** `kb/search_hybrid.py`: BM25Plus (стемминг [:6]) + dense v2,
   взвешенный RRF. Вес dense-ветки калиброван перебором 1.0/1.3/1.5/2.0 по шести
   запросам: при 1.0 лексический шум BM25 вытеснял лучшую dense-цитату Q6 из top-5,
   при **2.0** все отслеживаемые цитаты в top-5 без потерь лексических попаданий.
   `scripts/kb_query.py` использует гибрид по умолчанию (`--dense` — чистый dense).

Итог на тех же 6 запросах (гибрид, w=2.0): **цитируемость 4/6** (как у чистого
dense-v2), но качественно лучше: адресность Q6 усилилась (в топ пришла Усманова —
эталонный источник по шламам), в Q7 появился Манцевич-2008, служебный мусор из
топов исчез полностью. Q1 (насадки ГЦ) — подтверждённая дыра КОРПУСА: лексики
«песковая насадка» нет ни в одном документе, никакой поиск это не исправит —
нужен профильный источник по гидроциклонам. Q4 остаётся пограничным: механизм
шламовых потерь цитируется (Глембоцкий, с. 170), но точной связки «тонкие классы →
потери Ni» в top-5 нет.
